# v3.9.1 — Arctic 임베딩 서버 메모리 안정화 (누수 차단 + activation peak 완화)

## Summary

Arctic-ko 임베딩 서버(`localhost:8081`)의 두 메모리 문제를 수정했다. 둘 다 mlx 런타임의
메모리 거동에서 비롯하며, **회수 정확도·동작에는 영향 없는 순수 운영 안정화**(off-path).

1. **buffer cache 누수 차단** — mlx buffer cache 가 요청마다 가변 seq_len 버퍼를 무제한
   캐싱해 장시간 가동 시 `phys_footprint` 가 **24GB 까지 누수**(정상 baseline 748MB).
   32GB 머신에서 swap 포화 → 시스템 버벅임·팬. `mx.set_cache_limit(512MB)` 로 상한.
2. **긴 입력 activation peak 완화** — 8192토큰 단일 요청이 self-attention activation 으로
   순간 **~9GB peak** → 메모리 압박 시 OOM(프로세스 강제종료, 부하검증서 재현).
   `MAX_TOKENS` 8192→4096(O(n²)라 peak ~1/4) + 큰 입력(>1024tok) 후 `mx.clear_cache()`.

## Changed

- **EDIT** `scripts/arctic_ko_server.py`
  - `mx.set_cache_limit(512 * 1024 * 1024)` — buffer cache 상한 (#1)
  - `MAX_TOKENS` 8192 → 4096 — activation peak ~1/4 (#2)
  - `embed()` 후 입력 >1024tok 시 `mx.clear_cache()` — peak 잔여 즉시 반환, 연속 큰 요청 OOM 누적 방지 (#2)
  - `/health` 에 `cache_mb`/`active_mb` 노출 — 누수 재발 조기감지

## 검증

- **누수(#1)**: 운영 입력 분포 600회 부하 → phys 24GB→**1.3GB 평탄**(우상향 0), cache 512MB
  상한 유지, fails 0/600.
- **peak(#2)**: 8192때 9.3GB peak + OOM 크래시 났던 가혹 테스트(랜덤 ~28000자 300회)를 4096
  으로 재실행 → peak max **2.0GB**(약 1/4.6), 크래시 0, fails 0.
- 실측(메모리 282개) p95=3743tok → MAX_TOKENS 4096 에서 **95% 무손실**, 초과 9개만 추가
  truncate(정확도 trade-off 수용).
